Goal

Option D — The Winner
Dense SKU grid + sum-then-match
Register SKUs at every multiple of 5.
Frontend sums base + selected modifiers.
Find the exact SKU matching that total.
One useBits() call. Zero premium.
WORKS
Key insight: different modifier combos
that sum to the same number reuse ONE SKU.

Why it's not factorial — the "collapse"

2⁷ = 128 on/off modifier combinations, but their sums overlap heavily. Multiple combos producing the same total reuse a single SKU.

Modifier-sum 15: "KB Stick(15)" OR "Baby(5) + Speed(10)" Modifier-sum 25: "Big(25)" OR "Tiny(25)" OR "Baby+Sharp" OR "Baby+KB+..." (4 combos!) Modifier-sum 35: "Speed+Big(10+25)" OR "Speed+Tiny" OR "KB+Sharp(15+20)" OR ... (4 combos!) Modifier-sum 50: "Diamond(50)" OR "Big+Tiny+..."(multiple) (many!) 128 combinations → only 31 distinct sums (0, 5, 10, 15, ... 150)

4. The Quantity Question

Yazan's idea also included spawning multiple mobs (e.g. 3 silverfish). This created a new pricing problem.

Problem
"Extra mob" value scales with base price
Fixed-cost modifier: +25 bits on a 10-bit item = scam;
+25 bits on a 200-bit item = steal.
No single number works across the range.
↓

Percentage-based (+50% of base)

  • Base 25 → +12.5 → not multiple of 5
  • Breaks the exact-match SKU guarantee
  • Reintroduces premiums / snapping
BREAKS GRID MATH

Dynamic tiers (viewer picks 1/3/5)

  • 1× / 2× / 4× multipliers (clean on grid)
  • Tiers + modifiers on bulk = 250+ SKUs
  • FAILS at base ≤300 (276 total)
  • Fits only at base ≤75 (73 SKUs)
TIGHT / TINY BASE CAP

Streamer sets quantity, viewer adds modifiers

  • Quantity is part of item config (like now)
  • Viewer still picks modifiers dynamically
  • 73 SKUs, base ≤300 — clean fit
  • Edge case: bulk items priced >300 lose modifiers
EDGE CASE

Raise cap to 500 + streamer sets quantity

  • Streamer sets quantity AND base price
  • Viewer adds modifiers dynamically
  • "5 big fast zombies" IS possible
  • 112 new SKUs, 138 total, 112 free
  • Preserves the fun combo

Revenue reality check (the "discount" fear)

The "lost revenue" from bulk is mostly theoretical — viewers won't do 5 separate transactions. The real choice is between 1 mob vs. a bulk bundle.

6. The Two Remaining Options

Option 1 — Cap 300, streamer sets qty

  • 73 new SKUs registered (every multiple of 5, 5–450)
  • 99 total SKUs, 151 free for future features
  • Streamer configures quantity per item (1, 3, 5...)
  • Viewer adds modifiers dynamically (7 options)
  • Base price cap: 300 bits for modifier-enabled items
  • Edge case: bulk items >300 bits = no modifiers
  • Most SKU headroom for future (reverb on sound, etc.)

Option 2 — Cap 500, streamer sets qty

  • 112 new SKUs registered (every multiple of 5, 5–650)
  • 138 total SKUs, 112 free for future features
  • Streamer configures quantity per item (1, 3, 5...)
  • Viewer adds modifiers dynamically (7 options)
  • Base price cap: 500 bits for modifier-enabled items
  • "5 big fast zombies" at base 400 IS possible
  • Preserves the feature's most fun combination
  • Still plenty of SKU headroom

8. Example Purchase Flows

Flow 1 — Simple: Base 100 zombie + Big(25) + Speed(10) = 135 → useBits("value135") → 1 big fast zombie
Flow 2 — Full punish (the dream): Base 400 "5 zombies" item + Big(25) + KB Stick(15) + Diamond(50) = 490 → useBits("value490") → 5 big zombies with knockback sticks and diamond armor
(only possible with cap 500; requires base ≤300 otherwise)
Flow 3 — Yazan's original idea: Base 25 + Tiny(25) + KB Stick(15) = 65 → useBits("value65") → 1 tiny zombie that punts you
Flow 4 — No modifiers: Base 50 "3 zombies" item, viewer adds nothing → 50 → useBits("value50") (already exists!) → 3 plain zombies
